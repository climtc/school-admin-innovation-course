# M3 과제 명세 — 나의 학교 업무분장 구조 분석

> M3 · 7주차 안내 | 스키마 원천: 업무사건 역할 필드 [kb://design/dataset_design_doc] | 검증 근거: 교사 검증 카드팩 [kb://validation/teacher_cards] | 데이터 비교: 연간 업무사건 후보 714건 [kb://calendar/work_event_candidates_714]

## 과제 개요

M1에서 만든 "나의 학교 업무 인벤토리"를 한 단계 확장합니다. 이번에는 자기 소속 학교의 업무 **5건**을 골라, 각 업무를 **업무 → 주 담당 → 협업자 → 검토 경계**의 구조로 분석합니다. M1이 "무엇이 발생하는가", M2가 "왜 발생하는가"였다면, 이번 과제는 **"왜 그가 담당하고, 그중 어디까지 AI가 도울 수 있는가"**입니다. 이 분석은 M5의 AI 활용 전략과 M6 혁신 계획서에서 "우리 학교 적용" 부분의 핵심 근거가 됩니다.

M1 인벤토리에서 고른 5건을 그대로 가져와도 좋고, 담당 구조가 더 흥미로운 업무로 새로 골라도 좋습니다. 가능하면 협업자가 여럿인 업무(현장체험학습·방과후·늘봄·평가·운영위원회 등)를 1건 이상 포함하세요 — 담당 구조 분석이 풍부해집니다.

## 제출 형식 — 분석 표 (5건 모두 작성)

각 업무를 다음 열로 갖는 표를 만드세요. 5행입니다.

| 열 | 작성 요령 |
|---|---|
| `task_name` (업무명) | 일반 유형명으로. 학교 고유 행사명·프로그램명 변환 (예: "○○제 운영" → "학교 축제 운영") |
| `domain` (도메인) | 10개 도메인 중 1개 |
| `actor_primary` (주 담당) | 역할명만 — 담당교사, 부장교사, 교무행정사, 행정실 등 (실명 금지) |
| `actor_collaborators` (협업자) | 역할·기관 유형만 — 관리자, 행정실, 학부모, 위원회, 교육(지원)청, 외부업체 등 |
| `why_this_actor` (담당 근거) | 왜 이 역할이 주 담당인가 1문장 (도메인·시스템·산출물 근거로) |
| `review_boundary` (검토 경계) | 일곱 영역 중 걸리는 것 모두: 개인정보 / 안전 / 예산 / 계약 / 평가 / 민원 / 최종승인 |
| `ai_level` (허용 수준) | 이 업무의 핵심 작업에 draft / verify / recommend / assist_only / not_allowed 중 1개 + 근거 1문장 |

추가로, 5건 중 **협업자가 가장 많은 1건**을 골라 **단계별 표**(읽기자료·7주차 실습의 5단계 방식)를 1개 더 붙이세요. 그 1건이 "한 건"이라는 셈법에 가려진 반복·조율 부담을 어떻게 담고 있는지 단계로 펼쳐 보입니다.

## 비식별 체크리스트 (제출 전 전부 확인 — 하나라도 위반 시 재제출)

- [ ] 학교명이 없다 (학교급·규모 일반 정보는 가능 — 예: "중규모 초등학교")
- [ ] 교사·학생·학부모·외부인의 실명이 없다 (역할명만 사용)
- [ ] **담당자를 역추적할 수 있는 단서가 없다** — 작은 단위에서 "특정 업무 담당"이 곧 특정 개인을 가리키지 않도록, 직위·담당명을 일반 역할명으로만 표기
- [ ] 지역명·교육청명이 없다 ("소속 교육청", "관할 교육지원청"으로 일반화)
- [ ] 특정 업체명·기관명·은행명이 없다 ("외부업체", "위탁업체" 등으로 일반화)
- [ ] 특정 학생 사안(학교폭력·상담·특수교육 대상자 등)을 역추적할 날짜·사건 세부가 없다 (절차 수준으로만)
- [ ] 문서번호·공문번호·계약번호가 없다

## 에이전트(AI 비서) 활용

로컬 비서에게 담당 가설과 허용 수준 배정 초안을 맡길 수 있습니다. 단 세 가지를 지키세요. 첫째, **담당 근거를 반드시 확인**하고 동의 여부를 본인이 판정할 것(표의 `why_this_actor`는 본인 명의의 판단입니다). 둘째, 검토 경계·허용 수준 배정의 근거를 두 읽기자료의 정의와 대조해 직접 검산할 것. 셋째, **개인정보·민원·안전·평가가 걸리는 업무는 비식별 처리 전의 기록을 외부 서버형 AI에 입력하지 말 것**(민감도 라우팅 — 로컬에서만 다룹니다).

## 평가 루브릭 (4기준 × 3수준)

| 기준 | 미흡 (1) | 충족 (2) | 우수 (3) |
|---|---|---|---|
| **담당 구조 분석** — 주 담당·협업자·근거 | 담당/협업 구분이 모호하거나 `why_this_actor`가 2건 이상 비어 있음 | 5건 모두 주 담당·협업자가 구분되고 근거가 있으나 일부가 "원래 그래서" 수준 | 5건 모두 담당 근거가 도메인·시스템·산출물과 연결되고, 협업 관계의 의미(조율 부담)까지 짚음 |
| **검토 경계 식별** — 일곱 영역 적용 | 검토 경계 표시가 누락되거나 명백히 부적절한 건이 2건 이상 | 모든 건에 검토 경계가 표시되고 대체로 타당하나, 복수 경계가 걸리는 건을 단순화함 | 복수 경계가 걸리는 건을 정확히 식별하고, 그것이 단일 허용 수준으로 다룰 수 없음을 설명함 |
| **허용 수준 배정** — 5단계 + 근거 | 허용 수준이 없거나 근거 없이 배정됨 | 모든 건에 수준과 근거가 있으나 검토 경계와의 연결이 약함 | 수준이 검토 경계·민감도 라우팅과 연결되고, P-A 원칙(사람의 확정 행위)이 함께 명시됨 |
| **비식별 준수** — 체크리스트 + 역추적 방지 | 체크리스트 위반 또는 담당자 역추적 단서가 1개 이상 | 체크리스트를 통과하나 일반화가 과도해 담당 구조가 잘 안 읽히는 건이 있음 | 체크리스트를 통과하면서도 담당·협업·경계 구조가 온전히 읽힘 — 비식별과 정보성의 균형 |

판정: 12~10점 우수 / 9~7점 충족 / 6점 이하 재제출. 단, **비식별 준수가 '미흡'이면 총점과 무관하게 즉시 재제출**입니다(담당 구조 분석은 역추적 위험이 가장 큰 과제입니다).

## 제출

- 분량: 분석 표 1개(5행) + 협업자 최다 1건의 단계별 표 1개 + 담당 근거 5문장
- 기한: M4 첫 수업(8주차) 시작 전
- 8주차 도입에서 동료와 분석을 교환해 ①담당 근거의 타당성 상호 검토 ②"우리 학교에서 한 역할에 업무가 몰리는 구조"에 대한 토론을 진행하고, 그 결과를 M4 파이프라인 설계의 출발점으로 삼습니다.
