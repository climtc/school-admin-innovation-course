# M1 과제 명세 — 나의 학교 업무 인벤토리

> M1 · 2주차 안내 → 3주차 제출·공유 | 스키마 원천: 업무사건 추출 필드 [kb://design/dataset_design_doc] | 분류 기준: 10개 도메인 [kb://calendar/domain_summary]

## 과제 개요

자기 소속 학교에서 최근에 수행했거나 진행 중인 **행정업무 5건**을 대표학교 데이터셋과 같은 업무사건 스키마로 기록합니다. 이 인벤토리는 M2(발생 원인 분석), M3(담당 구조 분석)에서 계속 확장하며, M6의 혁신 계획서에서 "우리 학교 적용" 부분의 기초 자료가 됩니다. 즉, 한 번 내고 끝나는 과제가 아니라 **코스 내내 자라는 자료의 첫 버전**입니다.

## 기록 스키마 (5건 모두 다음 필드를 채울 것)

| 필드 | 작성 요령 |
|---|---|
| `task_name` (업무명) | 자연어로. 식별 정보 제거 후 일반화 — 예: "급식물품 구매 계약", "학교자율시간 운영계획 수립" |
| `domain` (도메인) | 10개 도메인 중 1개: 급식보건안전 / 평가교육과정 / 개인정보민원보고 / 방과후늘봄 / 학교운영위원회 / 행정재정 / 학교폭력생활지도 / 특수복지 / 학교구조변수 / 미분류. **분류 근거 1문장 필수** |
| `trigger` (트리거) | 무엇이 이 업무를 발생시켰는가 — 예: 학기 시작, 상급기관 공문 접수, 정산 시점 도래, 사안 발생, 위원회 개최 주기 |
| `season_or_month` (시기) | 월 또는 상시 |
| `actor_primary` (주 담당) | 역할명으로만 — 담당교사, 부장교사, 교무행정사, 행정실 등 (실명 금지) |
| `actor_collaborators` (협업자) | 역할·기관 유형으로만 — 관리자, 행정실, 학부모, 교육(지원)청, 위원회, 외부업체 등 |
| `related_systems` (시스템) | NEIS, K-에듀파인, 업무관리, 학교알리미, 학교 홈페이지, 알림 앱(일반 명칭) 등 |
| `output_documents` (산출문서) | 계획서, 가정통신문, 정산서, 회의록, 결과보고 등 문서 유형명 |
| 메모 (선택) | 병목·부담 지점 한 줄 — M2 이후에 쓸 단서 |

작성 형식: 위 필드를 열로 갖는 표(스프레드시트 또는 마크다운 표) 1개. 5행.

## 5건 선정 가이드

- 가능하면 서로 다른 도메인에서 고르세요(최소 2개 도메인 이상).
- "데이터에 없는 내 업무"(2주차 실습 단계 3에서 찾은 것)를 1건 이상 포함하면 좋습니다 — 가시성 결손 토론이 풍부해집니다.
- 수업·생활지도 자체가 아니라 그에 수반되는 **행정업무**(문서·수합·보고·정산·시스템 입력)를 고르세요.

## 비식별 체크리스트 (제출 전 전부 확인 — 하나라도 위반 시 재제출)

- [ ] 학교명이 없다 (필요하면 "본교" 또는 "소속 학교"로 표기. 학교급·규모 일반 정보는 가능 — 예: "중규모 초등학교")
- [ ] 교사·학생·학부모·외부인의 실명이 없다 (역할명만 사용)
- [ ] 지역명·교육청명이 없다 ("소속 교육청", "관할 교육지원청"으로 일반화)
- [ ] 특정 업체명·기관명·은행명이 없다 ("급식 납품업체", "위탁업체", "거래 은행" 등으로 일반화)
- [ ] 특정 학생 사안(학교폭력, 상담, 특수교육 대상자 등)을 역추적할 수 있는 날짜·사건 세부가 없다 (사안 업무는 절차 수준으로만 기술)
- [ ] 문서번호·공문번호·계약번호가 없다
- [ ] 업무명에 학교 고유 행사명·프로그램명이 그대로 들어 있지 않다 (일반 유형명으로 변환 — 예: "○○제 운영" → "학교 축제 운영")

## 에이전트(AI 비서) 활용

"내 업무 5건을 10개 도메인으로 분류해줘"라고 로컬 비서에게 맡길 수 있습니다. 단 두 가지를 지키세요. 첫째, **분류 근거를 반드시 확인**하고 동의 여부를 본인이 판정할 것(과제에는 본인 명의의 분류 근거 1문장이 들어갑니다). 둘째, 비식별 처리 **전의** 업무 기록(실명·학교명 포함)은 외부 서버형 AI에 입력하지 말 것 — 비식별이 끝난 버전만 다루게 하세요.

## 평가 루브릭 (4기준 × 3수준)

| 기준 | 미흡 (1) | 충족 (2) | 우수 (3) |
|---|---|---|---|
| **스키마 충실성** — 필수 필드 작성 | 필드 누락이 2건 이상의 업무에서 발생하거나 형식이 스키마와 다름 | 5건 모두 필수 필드가 채워져 있으나 일부 항목이 모호함(예: 트리거를 "해야 해서"로 기술) | 5건 모두 필드가 구체적으로 채워지고, 트리거·산출문서가 실제 업무 흐름을 추적 가능하게 기술됨 |
| **분류 타당성** — 도메인 분류와 근거 | 분류 근거가 없거나, 분류가 명백히 부적절한 건이 2건 이상 | 모든 건에 분류 근거가 있으며 대체로 타당하나, 경계 사례에 대한 검토가 얕음 | 분류 근거가 분류체계의 정의와 연결되고, 경계 사례(미분류 포함)에 대해 대안 분류까지 검토함 |
| **비식별 준수** — 체크리스트 이행 | 체크리스트 위반 항목이 1개 이상 존재 | 체크리스트를 모두 통과하나 일반화 수준이 과도해 업무 내용 파악이 어려운 건이 있음 | 체크리스트를 모두 통과하면서도 업무의 구조(트리거·담당·시스템)는 온전히 읽힘 — 비식별과 정보성의 균형 |
| **분석적 성찰** — 선정과 메모의 깊이 | 5건이 단일 도메인에 치우치고 메모가 없음 | 2개 이상 도메인에서 선정하고 메모가 일부 있음 | 도메인이 다양하고, "데이터에 없는 업무" 1건 이상 포함, 메모가 병목·부담을 구체적으로 짚어 M2의 질문으로 이어짐 |

판정: 12~10점 우수 / 9~7점 충족 / 6점 이하 재제출. 단, **비식별 준수가 '미흡'이면 총점과 무관하게 즉시 재제출**입니다.

## 제출

- 분량: 표 1개(5행) + 분류 근거 5문장 + (선택) 메모
- 기한: 3주차 수업 시작 전
- 3주차 수업에서 동료와 인벤토리를 교환해 ①분류 타당성 상호 검토 ②"데이터에 안 보이는 업무" 수집 토론을 진행합니다.
