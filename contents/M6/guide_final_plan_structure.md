# 가이드 — 혁신 계획서 구성 (M1~M5 누적 산출물 통합)

> M6 · 13주 가이드 | 자원: `kb://whitepaper/stage1 ~ stage9` · `kb://validation/ai_strategy_packet` · `kb://simulator/app` · `kb://calendar/work_event_candidates_714`

## 이번 가이드의 목적

M1~M5에서 만든 산출물(업무 인벤토리 → 원인 분석 → 담당 구조 → 파이프라인 설계 → AI 전략)을 하나의 **"AI 활용 행정업무 혁신 계획서"**로 통합하는 방법을 안내한다. 새로 쓰는 게 아니라, 다섯 모듈의 산출물을 엮는 작업이다.

## 1. 누적 산출물 지도

| 모듈 | 산출물 | 계획서에서의 위치 |
|---|---|---|
| M1 | 업무 인벤토리 5건 | 현황 진단(무엇이 발생하는가) |
| M2 | L0–L4 원인 분석 | 원인 진단(왜 발생하는가) |
| M3 | 담당 구조·검토 경계 | 책임 구조(왜 그가 담당하는가) |
| M4 | 데이터 파이프라인 설계 | 실행 기반(데이터를 어떻게 다루나) |
| M5 | AI 활용 전략표 | 혁신 방안(AI를 어떻게 쓰나) |

이 다섯이 "진단 → 원인 → 책임 → 기반 → 방안"의 논리적 흐름을 이룬다. 계획서는 이 흐름을 따른다.

## 2. 백서 9단계 골격 참조

계획서의 실행 단계는 백서 9단계 카드[kb://whitepaper/stage1 ~ stage9]의 구조를 참조한다 — 데이터 수집 → 전처리 → 스킬·하네스 → 시뮬레이터/앱 → 근거 기반 생성 → 배포 → 사용자 검증 → 분석 → 피드백의 폐루프. 자기 학교 혁신 계획을 이 9단계에 매핑하면, "한 번 하고 끝"이 아니라 **검증과 개선이 도는 계획**이 된다.

## 3. 계획서 권장 구성

1. **요약.** 혁신의 핵심 한 단락.
2. **현황 진단.** M1 인벤토리 기반, 우리 학교 업무 부담의 실태.
3. **원인·책임 분석.** M2·M3 기반, 왜 발생하고 왜 그가 담당하는가.
4. **데이터 기반.** M4 파이프라인 설계 요약(비식별·윤리 포함).
5. **AI 활용 방안.** M5 전략표 — 업무별 자동화 수준·검토 경계·데이터 처리 위치.
6. **실행 로드맵.** 백서 9단계에 매핑한 단계별 실행·검증 계획.
7. **거버넌스.** 사람 검토 경계, 데이터 주권("원데이터는 기기에"), 동의·감사 장치.

## 4. 주의 — 계획서가 지켜야 할 두 원칙

- **데이터 주권.** 민감 데이터는 기기 내 처리를 기본으로(M5 전략과 일관).
- **사람 중심.** AI 허용 수준이 높은 업무(not_allowed·assist_only)는 사람이 지킨다는 점을 명시.
