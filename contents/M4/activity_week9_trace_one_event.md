# 9주차 실습 — 업무사건 1건을 원천 문서까지 역추적하기

> M4 · 9주 실습 안내 | 자원: `kb://pipeline/extract_work_events` · `kb://design/dataset_design_doc` · `kb://calendar/work_event_candidates_714`

## 실습 목표

추출된 "업무 사건" 한 건을 골라, 그것이 어느 원천 문서에서 어떻게 만들어졌는지 거꾸로 따라간다. 파이프라인의 추적성(traceability)을 몸으로 확인하고, "데이터가 곧 진실"이 아니라 "데이터는 흔적의 재구성"임을 이해하는 것이 목적이다.

## 준비물

- 연간 업무사건 후보 데이터[kb://calendar/work_event_candidates_714]에서 한 건 선택
- 도메인 분류체계 설계서[kb://design/dataset_design_doc]
- 로컬 AI 비서(코스 앱)

## 단계

1. **사건 선택 (10분).** 자기 직무와 가까운 업무 사건 1건을 고른다. 그 사건의 필드(도메인·트리거·담당·시스템·산출문서)를 적는다.
2. **역추적 (30분).** 그 사건이 어떤 원천 문서에서 비롯되었는지 추적한다. 추출 로직[kb://pipeline/extract_work_events]이 어떤 텍스트 단서(키워드·문서 유형)로 이 사건을 식별했을지 추론한다.
3. **빈틈 식별 (20분).** 원천 문서가 말하지 않는 부분(시스템 내부 처리, 비공개 절차)을 표시한다. 이것이 M2에서 배운 "가시성 결손"의 구체적 사례다.
4. **AI 보조 (15분).** 로컬 비서에게 "이 업무 사건의 트리거와 담당을 도메인 분류체계로 검증해줘"라고 요청한다. **AI의 분류 근거를 반드시 확인**하고, 원천 문서와 대조한다.

## 분석 질문 5

1. 선택한 사건의 트리거는 무엇이며, 그것은 원천 문서의 어느 부분에서 확인되는가?
2. 이 사건은 단일 문서로 복원되는가, 여러 문서를 엮어야 하는가? 왜인가?
3. 추출 로직이 놓쳤을 법한 같은 도메인의 다른 사건이 있는가?
4. 원천 문서가 비식별 처리되어야 할 정보를 담고 있었는가? 파이프라인은 그것을 어떻게 다뤘어야 하는가?
5. 이 역추적이 불가능했다면(추적성이 없었다면) 분석의 신뢰도는 어떻게 달라지는가?

## 산출물

역추적 1건 기록표(사건 필드 → 추정 원천 단서 → 가시성 결손 표시 → AI 검증 결과)를 제출한다. 비식별: 특정 학교·인명·지역을 적지 않는다.
