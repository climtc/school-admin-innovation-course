# J8 — 코스 지식베이스 허브 아키텍처: 자원 포괄 분석과 최적 설계

> **버전.** v1.0 (2026-06-08) · **상위.** J7(PoC 계획) · **실행 산출물.** `Codex-Project/school-admin-innovation-course/` (깃헙 공개 저장소)
> **작성 방식.** 병렬 에이전트 하네스 — 리서치 2팀(R1·R2) → 공유 스펙 고정 → 작성 3팀(W-A 사람용 실라버스 / W-B M1 콘텐츠 / W-C 에이전트 인터페이스) disjoint 경로 동시 생산 → 기계 검증 통과.

---

## 0. 쉬운 설명 (Plain Explanation)

사용자의 전망은 이랬다 — "Codex가 만드는 파인튜닝용 데이터셋이 서버 측 실라버스·콘텐츠·에이전트 자원의 지식베이스가 될 것이다." 방향은 맞지만 화살표가 반대다. **파인튜닝 데이터셋이 지식베이스가 되는 것이 아니라, 하나의 지식베이스에서 파인튜닝 데이터셋을 포함한 네 가지 산출물이 모두 파생되어야 한다.** 그래야 실라버스가 말하는 내용, 에이전트가 아는 내용, RAG가 검색하는 내용, 파인튜닝된 모델이 학습한 내용이 서로 어긋나지 않는다. Codex가 지금 만드는 전처리·정제 산출물은 이 지식베이스의 *입력*으로 받아들이고, instruction 쌍은 네 파생물 중 하나로 위치를 다시 잡는다.

## 1. 자원 포괄 분석 (2026-06-08 기준)

| 계층 | 자원 | 상태 | 역할 판정 |
|---|---|---|---|

| **정제(L1 입력)** | Codex 진행 중: 파인튜닝용 전처리·데이터셋 구축 | **진행 중** | 산출물 중 *정제 텍스트·구조화 레코드*는 L1 Course KB 입력으로 수용. *instruction 쌍*은 L2-d 파생물로 재분류 (§3 수용 계약) |
| **파생(L2)** | 이번에 생산: 사람용 실라버스 + M1 콘텐츠 + 에이전트 인터페이스 (저장소) | **완료 (v0.1.0)** | L2-a·L2-b 1차 버전 |

핵심 발견: 원천 자산은 코스 6모듈 전부를 실증 데이터로 뒷받침할 만큼 충분하다 (10개 도메인·714 업무사건·L0-L4 조건 320개·시뮬레이터 평가루프·백서 9단계·교사 검증팩). 부족한 것은 콘텐츠가 아니라 **일관성을 보장하는 구조**였다.

## 2. 최적 아키텍처 — Course Knowledge Hub (단일 출처, 4중 파생)

```
[L0 원천 연구 자산 (읽기 전용)]
        │  추출·비식별·버전 스탬프 (Codex 전처리 산출물 합류 지점)
        ▼
[L1 Course KB — 단일 진실 공급원]  kb_items.jsonl (id·type·text·source·hash·deid·module_tags)
        │
        ├─ L2-a 사람용: SYLLABUS_HUMAN.md + contents/M1..M6  (이번 산출)
        ├─ L2-b 에이전트용: syllabus_agent.json + interface.json + manifest + skills 5종  (이번 산출)
        ├─ L2-c RAG 인덱스: 로컬 앱이 kb_items에서 빌드 (W2)
        └─ L2-d SFT 데이터셋: kb_items에서 생성된 instruction 쌍 (Codex 산출물 합류, W3)
        │
        ▼
[L3 소비자]  서버 허브(서명 배포) → 로컬 앱(RAG+스킬+라우팅) → 파인튜닝(MLX-LM) → 평가(2×2)
```

**규칙 3개.**
1. **파생물끼리 직접 참조 금지.** L2-a가 L2-d를 인용하거나 그 역은 금지 — 모든 일관성은 L1 경유로만. (cascade-update 가능 조건)
2. **resource_id가 공용어.** 사람 문서·에이전트 JSON·RAG 청크·SFT 메타데이터가 같은 `kb://` ID를 쓴다 — 이번 산출물에서 이미 시행 (manifest 22항목).
3. **버전 동반.** 모든 파생물은 KB 버전을 스탬프 (course_package v0.1.0 ↔ KB v0.1.0 ↔ adapter v… — 평가 리포트에서 추적).

## 3. Codex 산출물 수용 계약 (도착 시 적용)

| Codex 산출물 유형 | 수용 위치 | 조건 |
|---|---|---|
| 정제 텍스트·구조화 레코드 (문서→텍스트, 업무 레코드) | **L1 kb_items.jsonl로 변환 합류** | id·source_path·sha256·deid_status 필수, module_tags 부여 |
| instruction 쌍 (Q&A, 분류, 피드백 스타일) | **L2-d sft/train.jsonl** | held-out 분리 유지, 원천 kb_item id 역참조 필수 |
| 평가셋 | L2-d sft/eval/ | 학습셋과 원천 문서 수준 분리 검증 (J7 §6.4) |

검수 절차: 도착 → 스키마 검증(json) → 비식별 grep → 표본 10% 사람 검수 → KB 버전 증가 → 영향 파생물 재빌드 목록 산출.

## 5. 개발 전략 (다음 단계 — J7 스프린트에 정렬)

1. **S1 잔여.** 서버 허브가 이 저장소를 course_package로 서빙 (git pull → 서명 → 배포). Codex 산출물 도착 시 §3 계약 적용 → KB v0.2 → 파생물 재빌드.
2. **S2.** 로컬 앱이 syllabus_agent.json + manifest를 로드 → 스킬 5종 구현(이미 사양 완료) → RAG 인덱스 빌드(L2-c) → base vs RAG 측정.
3. **S3.** L2-d로 MLX-LM 파인튜닝 (E4B 완주 → 12B QLoRA).
4. **S4.** 2×2 비교 — 네 파생물이 같은 KB 버전에서 나왔으므로 비교가 공정해짐 (이 구조의 실질 이득).
5. **콘텐츠 확장.** M2~M6 본문은 M1과 같은 하네스 패턴으로 라운드 추가 (스텁에 resource_id 매핑 완료 — 착수 비용 낮음).
