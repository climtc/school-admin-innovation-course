# week-planner — 주차별 학습 계획

## 목적
15주 코스 구조(M1: 1–3주, M2: 4–5주, M3: 6–7주, M4: 8–10주, M5: 11–12주, M6: 13–15주)와 활동 의존 관계를 근거로, 학생의 현재 진도에 맞는 주간 학습·과제 계획을 만든다. 현직교사의 학교 업무 부하 시기를 반영해 실행 가능한 계획을 우선한다.

## 입력
- 현재 주차 또는 목표 주차 구간
- 학생의 진도 상태 (완료한 activity_id 목록, 미완료·지연 항목)
- (선택) 학생의 가용 시간·학교 업무 일정 (입력 시 local 처리)

## 절차
1. `syllabus/syllabus_agent.json`에서 대상 주차가 속한 모듈의 activities·assessment를 읽는다.
2. 활동 의존 관계를 확인한다: M1-A3(자기 학교 업무 5건 기록)→M2-A2(L0–L4 매핑), M1~M5 산출물→M6-A1(계획서 통합), M6-A1→M6-A2(자가 점검)→M6-A3(동료평가)→M6-A4(발표). 선행 산출물이 없는 활동은 계획에서 선행 항목을 먼저 배치한다.
3. 주 1회 3시간 수업 외 자습·과제 시간을 활동별로 배분한다 (reading/discussion은 수업 내, fieldwork/writing/design은 수업 외 중심).
4. 학생이 학교 업무 일정을 제공했거나 요청하면 kb://calendar/monthly_domain_crosstab의 월별 피크를 참조하여 업무 폭증 시기(예: 학기 초·학년말)에 과제 부하가 겹치지 않게 조정안을 제시한다.
5. 각 계획 항목에 활동의 sensitivity·routing_hint를 표시하여, 학생이 어떤 작업을 로컬 환경에서 해야 하는지 미리 알게 한다.
6. 지연 학생의 경우 따라잡기 계획을 만든다: 의존 관계상 필수인 활동(선행 산출물 생산)을 우선하고, discussion 등 수업 내 활동은 요약 복습으로 대체 제안한다.
7. 계획 끝에 해당 주차의 제출물(assessment.artifact 연계)과 제출 전 점검(rubric-self-check 연계)을 리마인더로 붙인다.

## 사용하는 resource_id
- `kb://calendar/monthly_domain_crosstab` — 월별 업무 피크 기반 일정 조정
- `kb://whitepaper/stage1` ~ `kb://whitepaper/stage9` — M6 계획서 작성 단계 분해 (13–15주 역산 일정)
- `kb://simulator/app` — M5 실습 준비물 안내 (11–12주)
- `kb://design/dataset_design_doc` — M1 기록 과제 사전 준비 안내

## 라우팅·민감도 규칙
- 코스 구조만 다루는 일반 일정 계획은 normal·either.
- 학생의 개인 일정, 소속 학교의 내부 행사·업무 일정이 입력되면 sensitivity=sensitive, routing=local로 격상하고 해당 정보를 계획 산출물 외부로 내보내지 않는다.
- 계획에 포함된 sensitive 활동(M1-A3, M2-A2, M2-A4, M3-A2, M3-A4, M4-A3, M5-A3, M6-A1, M6-A2)은 로컬 작업임을 항상 표기한다.

## 출력 형식
```
[주차 계획] 주차별: 수업 내 활동 / 수업 외 과제 / 예상 소요 시간
[의존 관계] 선행 산출물 → 후속 활동 표시
[로컬 작업 표시] sensitive 활동 목록
[제출물 리마인더] 해당 주차 artifact + 제출 전 rubric-self-check 안내
```

## 금지사항
- 모듈·주차 매핑(스펙 §2)과 다른 임의 일정 생성 금지 — 조정은 과제 시간 배분 수준까지만.
- 활동 생략·면제 안내 금지 — 지연 시에도 대체·압축 제안까지만 하고 최종 결정은 교수자 몫으로 안내.
- 학생 개인 일정·학교 내부 일정의 저장·외부 전송 금지.
- 근거 없는 소요 시간 단정 금지 — 추정치임을 명시.
